9. El Rey de la Colina: Random Forest, XGBoost y LightGBM

En el capítulo anterior, rompimos las cadenas de la estadística univariante. Entendimos que para predecir la demanda real en un mercado globalizado y volátil, no basta con mirar la inercia del pasado; necesitamos integrar el contexto: promociones, cambios de precios, comportamiento de la competencia y variables climáticas. Pero antes de elegir un algoritmo al azar, debemos comprender las dos grandes filosofías de “ensamble” que dominan la ciencia de datos moderna: el Bagging y el Boosting.

Bienvenidos al territorio de los “pesos pesados”. En este capítulo, aprenderemos a elegir nuestra arma según la calidad de nuestros datos, la potencia de nuestro hardware y la estabilidad operativa que necesite nuestra red de suministro.

9.1 La Filosofía del Ensamble: La Sabiduría de la Multitud

En Machine Learning, un “ensamble” no es un modelo único, sino una combinación estratégica de muchos modelos base (generalmente cientos de árboles de decisión) diseñada para obtener una predicción mucho más robusta que la de cualquier individuo por separado. Es la aplicación práctica del teorema del jurado de Condorcet: si cada individuo tiene una probabilidad mayor al 50% de acertar, la probabilidad de que la mayoría acierte tiende al 100% a medida que añadimos participantes.

Existen dos formas fundamentales de construir este equipo de “expertos”:

A. Bagging y Random Forest: La Democracia Robusta del Bosque

El Bagging (Bootstrap Aggregating) es una técnica de paralelismo. Su exponente más famoso, el Random Forest, es considerado el “caballo de batalla” de la industria por su increíble resistencia al error.

  1. Bootstrap (Muestreo Aleatorio): El algoritmo crea cientos de versiones ligeramente diferentes de tus datos históricos mediante el muestreo con reemplazo. Cada árbol ve una “realidad” distinta: unos ven más promociones, otros ven más periodos de escasez.

  2. Inyección de Aleatoriedad: Además de los datos, el Random Forest solo permite que cada árbol elija entre un subconjunto aleatorio de variables en cada división. Esto evita que una variable muy dominante (como el precio) opaque otras señales sutiles pero importantes.

  3. Votación/Promedio: El resultado final es el promedio simple de todos los árboles.

  • Para el Arquitecto: El Random Forest es una máquina de reducir la Varianza. En Supply Chain, donde los datos suelen tener errores de digitación, picos por compras de pánico o ruidos inexplicables, el Random Forest actúa como un filtro natural. Es casi imposible de sobreajustar (overfitting) y es el modelo más seguro para automatizar productos de baja o media rotación.

B. Boosting: La Dictadura de la Mejora Continua

El Boosting es una técnica de secuencialidad. Es la arquitectura detrás de campeones como XGBoost y LightGBM. Aquí no hay democracia; hay un proceso de aprendizaje obsesivo y perfeccionista:

  1. El primer árbol realiza una predicción tosca (generalmente el promedio).

  2. El algoritmo calcula el residuo (el error exacto de esa predicción).

  3. El segundo árbol no intenta predecir la demanda; intenta predecir el error del primer árbol para corregirlo.

  4. Este proceso se repite cientos de veces, donde cada nuevo árbol se especializa en resolver los casos que sus predecesores fallaron.

  • Para el Arquitecto: El Boosting reduce el Sesgo (Bias). Es un “bisturí” estadístico increíblemente preciso capaz de detectar patrones de elasticidad-precio o efectos promocionales que el Random Forest ignoraría por ser “demasiado sutiles”. Sin embargo, es un arma de doble filo: si tus datos contienen “basura” o ruido no sistémico, el Boosting aprenderá a predecir esa basura de forma perfecta, fallando estrepitosamente en el futuro.

9.2 Duelo de Titanes de Boosting: XGBoost vs. LightGBM

Si tus datos son limpios, tus variables externas están bien registradas y buscas el menor error absoluto posible, entrarás en el mundo del Gradient Boosting. Aquí, la elección depende de la escala de tu operación:

XGBoost (eXtreme Gradient Boosting): El Veterano Confiable

Utiliza un crecimiento de árboles por niveles (Level-wise). Expande el árbol nivel por nivel de forma simétrica.

  • Fortaleza: Esta estructura lo hace extremadamente estable y permite un control muy fino de la complejidad mediante la regularización.

  • Ideal para: Datasets de tamaño medio donde la precisión es la prioridad número uno y el tiempo de entrenamiento no es una restricción crítica.

LightGBM (Light Gradient Boosting Machine): El Velocista de Microsoft

Utiliza un crecimiento por hojas (Leaf-wise). Identifica qué hoja específica (sin importar el nivel) reducirá más el error global y la expande primero. Además, utiliza una técnica llamada “Histogram-based learning” para agrupar valores de variables continuas.

  • Fortaleza: Es increíblemente rápido (hasta 10-15x más que XGBoost) y consume una fracción de la memoria RAM. Maneja de forma nativa variables categóricas (como el ID del SKU o la Ciudad), lo cual es una bendición para el analista de inventarios.

  • Ideal para: Catálogos masivos de Retail o E-commerce (+100k SKUs) donde los modelos deben re-entrenarse diariamente para capturar cambios bruscos en la demanda.

9.3 Simulador: ¿Promedio (Forest) o Corrección (Boosting)?

Usa este simulador para visualizar la diferencia fundamental en la “forma de pensar” de ambos modelos ante una demanda ruidosa.

  • Observa cómo el Random Forest (azul) crea una curva suave. Su objetivo es encontrar la “verdad central” ignorando los puntos que se desvían demasiado. Es un modelo de estabilidad.

  • Observa cómo el XGBoost (rojo) intenta “perseguir” los puntos. Con suficientes árboles, intentará tocar cada pico. Es un modelo de precisión extrema, pero requiere que confíes ciegamente en que esos picos no son ruido aleatorio.

jStat = require("jstat@1.9.6")
Plotly = require("plotly.js-dist-min@2.24.1")

viewof complexity_mode = Inputs.radio(["Random Forest (Estabilidad/Bagging)", "XGBoost (Precisión/Boosting)"], {value: "Random Forest (Estabilidad/Bagging)", label: "Arquitectura del Modelo:"})
viewof tree_count = Inputs.range([1, 200], {value: 20, step: 10, label: "Número de Árboles en el Ensamble:"})
viewof noise_level = Inputs.range([0, 60], {value: 25, step: 5, label: "Nivel de Ruido en la Demanda:"})
sim_ts = {
  const data = [];
  for (let i = 0; i < 100; i++) {
    const pattern = 120 + Math.sin(i/6) * 50 + (i * 0.3); // Ciclo + Tendencia
    const noise = jStat.normal.sample(0, noise_level);
    data.push({ t: i, y: Math.max(0, pattern + noise) });
  }
  return data;
}

// 2. Simulación de la lógica de aprendizaje de cada arquitectura
model_pred = {
  const y = sim_ts.map(d => d.y);
  let forecast = Array(y.length).fill(jStat.mean(y));
  
  if(complexity_mode.includes("XGBoost")) {
    // Simulación de Boosting: Ajuste secuencial de residuos
    const learning_rate = 0.1;
    for(let i=0; i < tree_count/2; i++) {
      for(let j=0; j < y.length; j++) {
        forecast[j] += (y[j] - forecast[j]) * learning_rate;
      }
    }
  } else {
    // Simulación de Bagging: Promedio local para reducir varianza
    const window_size = Math.max(1, 12 - (tree_count/20));
    forecast = sim_ts.map((d, i) => {
      const start = Math.max(0, i - Math.floor(window_size/2));
      const end = Math.min(y.length, i + Math.ceil(window_size/2));
      const window = sim_ts.slice(start, end).map(w => w.y);
      return jStat.mean(window);
    });
  }
  return forecast;
}

// 3. Renderizado del Gráfico Comparativo
{

}

9.4 El Factor Operativo: Escalabilidad y Mantenimiento

Como Arquitecto, no solo debes preocuparte por el error (MAE/WAPE), sino por la viabilidad operativa del modelo. Entrenar un modelo complejo para un solo SKU es fácil; hacerlo para 200,000 SKUs cada domingo por la noche es un desafío de ingeniería:

  1. Paralelismo: El Random Forest es naturalmente paralelo (puedes entrenar 100 árboles al mismo tiempo en 100 núcleos de CPU). El Boosting es secuencial (el árbol 2 necesita que el árbol 1 termine), lo que lo hace más lento de entrenar a menos que uses las optimizaciones de LightGBM.
  2. Sensibilidad a Outliers: Un Random Forest “perdona” un error en la data de entrada. Un XGBoost sin regularización intentará explicar ese error, distorsionando el pronóstico para los periodos siguientes.
  3. Variables Categóricas: Si tu base tiene muchas variables de texto (Color, Talla, Región), LightGBM las manejará de forma mucho más eficiente que XGBoost, reduciendo el tiempo de preparación de datos en un 60%.

9.5 La “Carrera de Caballos”: Selección por Torneo

En la Arquitectura de Suministros moderna, la mejor práctica no es elegir un algoritmo “por fe”, sino implementar un Torneo de Modelos (Tournament Selection):

  1. Para cada SKU (o familia), ejecutamos en paralelo un Random Forest, un LightGBM y un modelo estadístico (como el del Cap. 7).
  2. Utilizamos una técnica de Validación Cruzada Temporal (Rolling Window): entrenamos con los meses 1-24 y probamos en el mes 25. Luego entrenamos con 2-25 y probamos en el 26.
  3. El Ganador Toma Todo: El modelo que entregue el mejor MASE (Mean Absolute Scaled Error) en los periodos de prueba es el que se gana el derecho a generar la orden de compra real.

Este enfoque de “competencia de algoritmos” asegura que si un producto se vuelve ruidoso, el Random Forest gane el torneo; pero si el producto se vuelve altamente sensible a promociones, el Boosting tome el mando.
