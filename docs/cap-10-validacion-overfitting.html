10. La Carrera de Caballos: Selección Automática de Modelos

Has construido un arsenal formidable. Tienes desde la humilde simplicidad del modelo Naive hasta la potencia quirúrgica de XGBoost y la robustez democrática del Random Forest. Pero ahora te enfrentas al dilema del general: ¿A quién envío a la batalla?

En una operación real con 50,000 SKUs, no puedes elegir un modelo manualmente para cada uno. Intentar usar XGBoost para un producto con demanda errática es un desperdicio de recursos, y usar un Promedio Móvil para tu producto estrella es un suicidio financiero. En este capítulo, aprenderemos a diseñar la Arquitectura de Selección por Torneo.

10.1 La Ilusión de la “Talla Única”

El error más común en la implementación de sistemas de demanda es buscar el “Mejor Algoritmo”. No existe. La demanda es un camaleón que cambia según la categoría, la región y el ciclo de vida del producto.

La solución no es elegir un modelo, sino construir un Proceso de Competencia. En la arquitectura de suministros moderna, cada SKU corre su propio torneo interno cada vez que se actualizan los datos.

10.2 Anatomía de un Torneo (Tournament Selection)

El proceso de selección por torneo (o Champion-Challenger) sigue tres pasos rigurosos:

  1. Entrenamiento en Paralelo: El sistema entrena simultáneamente 3 o 4 arquitecturas distintas (ej. Holt-Winters, Random Forest y LightGBM) utilizando los datos históricos.

  2. Backtesting (La Prueba de Fuego): No medimos el error en los datos que el modelo ya vio. El sistema “esconde” los últimos meses de historia (Set de Validación) y le pide a los modelos que los predigan.

  3. Declaración del Ganador: El modelo que entregue el menor error en el set de validación es el que se gana el derecho de generar el pronóstico para el futuro real.

10.3 ¿Cómo elegir al Juez? Las Métricas de Selección

Aquí es donde retomamos nuestra auditoría del Capítulo 5. Para decidir quién gana el torneo, no usamos el MAPE. Usamos el MASE (Mean Absolute Scaled Error):

  • Si MASE < 1: El modelo es mejor que la estadística simple.

  • Si MASE > 1: El modelo está “alucinando”. El sistema debe descartarlo y volver a lo básico.

10.4 Simulador: El Torneo de Algoritmos

En este laboratorio, puedes ver una “Carrera de Caballos” en tiempo real. Hemos puesto a competir tres filosofías: Estadística (Holt-Winters), Bagging (Random Forest) y Boosting (XGBoost).

Ajusta el Ruido y la Complejidad de la demanda. Observa cómo el “Ganador” cambia: en escenarios ruidosos suele ganar la robustez del Forest, mientras que en escenarios con patrones claros, el Boosting es imbatible.

jStat = require("jstat@1.9.6")
Plotly = require("plotly.js-dist-min@2.24.1")

viewof noise_tournament = Inputs.range([5, 80], {value: 30, step: 5, label: "Ruido del Mercado (Varianza):"})
viewof pattern_complexity = Inputs.range([1, 5], {value: 2, step: 1, label: "Complejidad del Patrón:"})
data_tournament = {
  const data = [];
  const total_days = 100;
  for (let i = 0; i < total_days; i++) {
    const trend = i * 0.5;
    const season = Math.sin((i / 7) * 2 * Math.PI) * 40 * (pattern_complexity/2);
    const noise = jStat.normal.sample(0, noise_tournament);
    data.push({ t: i, y: Math.max(0, 100 + trend + season + noise), type: i < 86 ? 'train' : 'test' });
  }
  return data;
}

// 2. Ejecutar Motores en Paralelo (Simulación de lógicas)
tournament_results = {
  const train = data_tournament.filter(d => d.type === 'train');
  const test = data_tournament.filter(d => d.type === 'test');
  const last_train_y = train[train.length - 1].y;
  
  // Baseline Naive para MASE
  const naive_mae = jStat.mean(train.slice(1).map((d, i) => Math.abs(d.y - train[i].y)));

  const models = [
    { 
      name: "Estadística (H-W)", 
      forecast: test.map((d, i) => last_train_y + (i+1)*0.5 + Math.sin((d.t/7)*2*Math.PI)*20),
      color: '#94a3b8'
    },
    { 
      name: "Random Forest", 
      forecast: test.map((d, i) => jStat.mean(train.slice(-10).map(x => x.y))), // Suavizado
      color: '#2563eb'
    },
    { 
      name: "XGBoost", 
      forecast: test.map((d, i) => last_train_y + (i+1)*0.5 + Math.sin((d.t/7)*2*Math.PI)*40 + jStat.normal.sample(0, 5)),
      color: '#ef4444'
    }
  ];

  const processed = models.map(m => {
    const errors = test.map((d, i) => Math.abs(d.y - m.forecast[i]));
    const mae = jStat.mean(errors);
    const mase = mae / naive_mae;
    return { ...m, mae, mase };
  });

  const best_mase = Math.min(...processed.map(p => p.mase));
  return processed.map(p => ({
    ...p,
    winner: p.mase === best_mase
  }));
}

// 3. Gráfico del Torneo
{
  const container = html`<div style="width:100%; height:450px;"></div>`;
  const train = data_tournament.filter(d => d.type === 'train');
  const test = data_tournament.filter(d => d.type === 'test');
  
  const traces = [

  ];

  Plotly.newPlot(container, traces, {

    margin: {t: 50, b: 50, l: 50, r: 30}
  });
  return container;
}

10.5 Estrategia de Selección por Segmentación

Como Arquitecto, puedes pre-configurar el torneo basándote en tu segmentación ABC/XYZ:

  • Productos AX (Alta Importancia / Predecibles): El torneo debe ser riguroso. Incluye ARIMA, Holt-Winters y XGBoost. El costo de fallar aquí es enorme.

  • Productos CZ (Baja Importancia / Erráticos): No pierdas tiempo computacional. El torneo solo debe incluir Naive y un Promedio Móvil simple. Si la matemática compleja no le gana al Naive por más de un 5%, quédate con lo simple.

TipLa Regla del 5%

En muchos centros de excelencia de Supply Chain, un modelo “Challenger” (el nuevo) solo reemplaza al “Champion” (el actual) si logra reducir el error en al menos un 5%. Esto evita cambios nerviosos en la cadena de suministro por mejoras marginales que podrían ser puro ruido.

10.6 Implementación: El “Model Picker” Automático

El paso final de la arquitectura de ML es automatizar este Model Picker. El software debe:

  1. Correr el torneo mensualmente.

  2. Almacenar el historial de quién ganó cada vez (Auditoría de Modelos).

  3. Si un modelo gana consistentemente durante 6 meses, fijarlo como el estándar para esa categoría.
