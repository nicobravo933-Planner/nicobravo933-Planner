8. Machine Learning: Más allá de la Autocorrelación

Felicidades. Has dominado los pilares de la estadística clásica. Sabes suavizar el ruido, capturar la tendencia y seguir el ritmo de la estacionalidad con modelos como Holt-Winters. Pero, como todo Arquitecto de Suministros, pronto notarás que la realidad de los mercados actuales tiene más dimensiones de las que una sola serie de tiempo puede contener.

¿Qué pasa cuando la demanda no sube por “inercia”, sino porque hubo una promoción agresiva de la competencia? ¿Qué pasa cuando un festivo móvil (como Semana Santa o el Día de la Madre) cambia de fecha y arruina tus promedios históricos? Aquí es donde dejamos atrás la estadística puramente univariante para entrar en la nueva frontera: El Machine Learning aplicado a la Demanda.

8.1 El Gran Salto: Univariante vs. Multivariante

Hasta ahora, nuestros modelos (ARIMA, SES, Holt-Winters) eran fundamentalmente univariantes. Esto significa que solo miraban el pasado de la propia demanda para predecir el futuro. Es una técnica poderosa, pero limitada: es como intentar conducir un camión en una carretera con curvas mirando exclusivamente por el espejo retrovisor. Si bien el pasado nos da una idea del camino, no nos avisa de los obstáculos que tenemos enfrente.

El Machine Learning (ML) nos permite pasar al modelo multivariante. En lugar de preguntar solo “¿Qué vendí antes?”, el ML construye una visión panorámica integrando el contexto. El ML no busca solo la “autocorrelación” (el parecido de hoy con el ayer); busca patrones de relación entre múltiples variables que coexisten en el tiempo.

El Contexto como Señal

En un modelo multivariante, la demanda es la variable dependiente, pero está “explicada” por un ejército de variables independientes:

  • Elasticidad de Precio: ¿A qué precio lo vendí y cómo afectó el volumen?

  • Eventos y Promociones: ¿Había un descuento activo o un banner en el sitio web?

  • Factores de Calendario: ¿Es un día de pago (quincena)? ¿Es víspera de un puente festivo?

  • Variables Exógenas: ¿Hubo una ola de calor que disparó la venta de bebidas? ¿Cómo está el tipo de cambio?

8.2 Ingeniería de Atributos: El Verdadero Ingrediente Secreto

En Machine Learning, el algoritmo (sea XGBoost, Random Forest o una Red Neuronal) es solo el motor. El combustible, y lo que realmente determina la victoria, es la Ingeniería de Atributos (Feature Engineering). Esta es la tarea creativa y técnica de transformar datos crudos en variables que el modelo pueda procesar para extraer conocimiento.

Categorías de Atributos Críticos para el Arquitecto:

  1. Lags (Retrasos Temporales): Son los valores de ventas de hace 1, 7, 14 o 30 días. Siguen siendo la señal más fuerte porque capturan el “momentum”. Un buen arquitecto también crea promedios móviles de estos lags (ej. el promedio de ventas de las últimas 3 semanas) para filtrar el ruido.

  2. Atributos de Calendario (Dummies): Transformamos fechas en números. ¿Es lunes (1) o domingo (0)? ¿Es la semana 52 del año? Estas variables ayudan al modelo a entender que los lunes siempre se vende más que los martes, sin que tengamos que programar esa regla manualmente.

  3. Variables de Interacción: A veces, el efecto de una promoción es mayor si ocurre en fin de semana. Multiplicar la variable is_promo por is_weekend crea un atributo de interacción que revela patrones complejos que un modelo lineal jamás vería.

  4. Atributos de Producto y Ubicación: Incluir la categoría del producto o la zona geográfica permite que el modelo aprenda comportamientos grupales. Si un nuevo producto no tiene historia, el modelo puede “pedir prestado” el comportamiento de productos similares.

8.3 El Fin del Mito de la “Caja Negra”

Un mito persistente entre los gerentes tradicionales es que el ML es una caja negra impredecible y peligrosa. En la arquitectura moderna, esto es totalmente falso. Gracias a técnicas de Interpretabilidad como los SHAP Values (Shapley Additive Explanations) o el Feature Importance, el ML es a menudo más transparente que los modelos estadísticos oscuros.

Podemos realizar una autopsia del pronóstico y decir con precisión: “Este pronóstico de 500 unidades se descompone así: 350 son la base histórica, 100 se deben a la promoción activa y 50 al efecto del clima cálido previsto”. Esta capacidad de explicación transforma al planificador de demanda de un “creyente en el modelo” a un “auditor de drivers de demanda”, permitiendo tomar decisiones de negocio informadas y no solo basadas en la fe.

8.4 Simulador: El Poder de los Atributos (ML vs. Estadística)

En este laboratorio, comparamos un modelo estadístico simple (Promedio Móvil) contra un modelo de Machine Learning idealizado que posee el atributo de Promoción.

Observa qué sucede cuando activas una Promoción: el modelo estadístico es reactivo; solo empieza a subir después de ver ventas altas, y lo que es peor, genera un exceso de stock (over-stock) al final de la promoción porque su promedio móvil tarda días en “olvidar” el pico de demanda. El ML, por el contrario, se anticipa y se apaga quirúrgicamente.

jStat = require("jstat@1.9.6")
Plotly = require("plotly.js-dist-min@2.24.1")

viewof promo_impact = Inputs.range([10, 150], {value: 70, step: 5, label: "Impacto de Promoción (Unidades):"})
viewof promo_day = Inputs.range([10, 45], {value: 25, step: 1, label: "Día de Inicio de Promoción:"})
viewof noise_ml = Inputs.range([0, 40], {value: 12, step: 2, label: "Ruido del Mercado (Varianza):"})
ml_series = {
  const data = [];
  const promo_duration = 7;
  for (let i = 0; i < 60; i++) {
    const base = 100;
    const is_promo = (i >= promo_day && i < promo_day + promo_duration) ? 1 : 0;
    const effect = is_promo * promo_impact;
    const noise = jStat.normal.sample(0, noise_ml);
    data.push({ t: i, y: Math.max(0, base + effect + noise), promo: is_promo });
  }
  return data;
}

// 2. Modelo Estadístico (Promedio Móvil 7 días)
stat_model = {
  const forecast = [];
  const window_size = 7;
  for (let i = 0; i < ml_series.length; i++) {
    if (i < window_size) {
      forecast.push(100);
    } else {
      const window = ml_series.slice(i - window_size, i).map(d => d.y);
      forecast.push(jStat.mean(window));
    }
  }
  return forecast;
}

// 3. Modelo ML (Conoce la causa: Variable Binaria de Promo)
ml_model = {
  return ml_series.map((d, i) => {
    // El modelo ML "entiende" la relación estructural
    return 100 + (d.promo * promo_impact);
  });
}

// 4. Auditoría de Errores con enfoque FVA
ml_metrics = {
  const err_stat = jStat.mean(ml_series.map((d, i) => Math.abs(d.y - stat_model[i])));
  const err_ml = jStat.mean(ml_series.map((d, i) => Math.abs(d.y - ml_model[i])));
  const fva = err_stat - err_ml;
  return { err_stat, err_ml, fva };
}

ImportantEl Veredicto de la Inteligencia

  • Error Modelo Estadístico (MAE): unidades.

  • Error Modelo ML (MAE): unidades.

  • FVA (Forecast Value Added): unidades de mejora.

Lección para el Arquitecto: La estadística pura “persigue” el impacto de la promoción solo después de que ocurre, lo que genera quiebres de stock al inicio y excesos de inventario masivos cuando la promoción termina (el modelo sigue pidiendo porque el promedio móvil está inflado). El ML, al tener el atributo is_promo, entiende la temporalidad de la acción y ajusta el inventario de forma preventiva.

8.5 Implicaciones Estratégicas: ¿Cuándo NO usar ML?

A pesar de su potencia, el Machine Learning no es una solución mágica “instalar y olvidar”. El Arquitecto de Suministros debe evaluar los costos y riesgos operativos antes de la implementación:

  1. Hambre de Datos y Calidad Histórica: El ML necesita mucha historia para aprender. Pero más que cantidad, necesita calidad. Si no tienes un registro histórico limpio de cuándo fueron tus promociones pasadas o tus cambios de precio, el modelo no podrá establecer la relación causal. “Basura entra, basura sale”.

  2. El Problema del “Cold Start”: Para productos nuevos sin historia, un modelo multivariante basado en lags no tiene nada que procesar. Aquí dependemos de la inteligencia de atributos categóricos (ej. “se parece a este otro producto”).

  3. Costo de Mantenimiento: Entrenar y supervisar modelos para 100,000 SKUs requiere una infraestructura de datos (MLOps) y un equipo que sepa auditar los resultados. A veces, para productos Clase C (baja rotación), la estadística simple sigue siendo la decisión más rentable.

  4. Riesgo de Overfitting: Si le das 500 atributos a un modelo para predecir una serie de 24 meses, el modelo memorizará el ruido y fallará estrepitosamente en el futuro. Menos es más: selecciona solo los atributos que tengan una lógica de negocio detrás.
